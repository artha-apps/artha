**Defects found during the headed run — all fixed in-PR with tests:**
1. **ModelPicker stale chip** after onboarding — fixed (event subscription), regression-tested, re-verified headed.
2. **Row 13 blocker — zero-vector persistence** (founder-rejected as "warned ≠ prevented"): the indexer could still write fabricated 768-zero vectors. Fixed with the integrity patch (typed `EmbeddingUnavailableError`, `pending_embedding` state, retrieval exclusion, validated memory-embedding cache); 20 tests.
3. **Runtime status race** found by the row-10 stub: `model:ensure` returned the shared `lastStatus`, which a concurrent launch-time run overwrote, so onboarding showed "Starting your local model…" for a runtime that had already failed. `ensureModelReady` now returns its own terminal status; 5 tests.
4. **QA profile isolation leak** found during the row-13 rerun: `handlers.ts` captured the RAG indexer at module load — before `main.ts` applies `ARTHA_USER_DATA_DIR` — so an isolated session wrote its index file into the real profile directory. Content was synthetic-only and the live DB was unaffected (`getDb()` resolves lazily); the stray file was removed and the live profile verified clean (`integrity_check` ok, no `execution_profiles` table, no `rag-indexes` dir). Indexer is now resolved lazily, with a static regression guard.
5. Not product defects: the first-run feature tour intercepts programmatic input (driver-only), and L7 mock-only badge visibility (pre-recorded review finding, deferred).

**macOS keychain prompt investigation (corrects an earlier premature note).** Repeated authorization prompts during validation are caused by **ad-hoc signing of development builds**, not by production update behaviour:

| Build | Signature | Designated requirement | Stable across rebuilds? |
|---|---|---|---|
| Validation (`electron-builder --dir`) | `adhoc`, TeamIdentifier not set | `cdhash H"8f2685…"` — a hash of that exact binary | **No** — every rebuild changes it, so macOS treats each build as a different application and the keychain ACL no longer matches |
| Installed release | Developer ID Application (Team 6X83N47V3W) | `identifier "ai.artha.app" and anchor apple generic and … certificate leaf[subject.OU] = "6X83N47V3W"` | **Yes** — identity-based, not binary-hash-based |

Because the production designated requirement does not change when the binary changes, a correctly signed update is expected to retain keychain authorization. **Not characterized as production friction, and not a release blocker.** The empirical confirmation the founder specified (two consecutively built *Developer-ID-signed* packages, same bundle id/entitlements, realistic replacement path, sealed credential in a disposable profile) has **not** been run — recorded as a pre-release check, to be done alongside the next signed build.

**Profile-isolation audit (repo-wide).** Four userData-derived write roots exist, all resolved lazily at call time: `db/schema.ts` (`artha.db`, inside `initDatabase`), `rag/indexer.ts` (`rag-indexes`, memoized on first use), `ipc/handlers.ts` (`imported-bundles`, inside the handler), and `scheduler.ts` (read-only `statfs`). No module-level `app.getPath` binding, no hardcoded `Application Support/Artha` path (the only match is an unrelated `fnm` Node-version lookup), no direct reads of the QA env vars outside `system/qaProfile.ts`, and every service singleton (`MCPRegistry`, `SchedulerService`, `SkillRegistry`, `BrowserController`) constructs lazily via `getInstance()`. Skills/artifacts are DB-backed or written to user-chosen folders, not the profile root. Safeguards: the static CI guard plus a **runtime integration test** that performs real indexing writes and asserts every created file stays under the disposable root, that the production root is used unchanged without QA authorization, and that an override aimed at the live profile is refused.
